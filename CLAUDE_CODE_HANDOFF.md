## Что уже сделано (Срезы 1-5)

**Срез 1 — Result Page v2** (`/run/<id>/v2/`):

- KPI-группы 5 шт: Обзор · Энергобаланс · Фаза отката · Фаза наката · Конечное состояние
- Парные карточки min/max в фазах

- Все графики через единые хелперы `_apply_layout`, `_add_peak_marker`, `_add_recoil_vline` в `services/charting.py`
- Аннотации пиков фиксированно справа `ax=90, ay=0, arrowsize=1.4`

**Срез 2 — Дашборд** (`/`):
- 5 stat-карточек: Всего · За неделю · Успешных · С предупреждениями · В каталоге
- Фильтры (Все/Свежие/Успешные/С warning) + поиск + сортировка + пагинация
- Карточки расчётов с метриками
- Старая форма создания переехала на `/new/` (имя URL `index` сохранено)

**Срез 3 — Каталог тормозов** (`/catalog/`):
- Модель `BrakeCatalog` (миграция 0018), copy-on-use в расчёт
- CRUD: list / new / edit / delete / detail
- На detail для параметрических — таблица всех 12 параметров со символами и формулой `F_T = (B̄₃·k̄_B·Ȳ_a)² · V · [2θ_κ/R_κ + 4(2N-1)θ_y/R_y]`
- На detail для табличных — Plotly график F(v) inline + статистика + таблица точек
- В форме расчёта переключатель «✎ Вручную / ⊟ Из каталога» с автоподстановкой параметров и имени
- AJAX-эндпоинт `/catalog/save-from-form/` для сохранения тормоза прямо из формы расчёта

**Срез 4 — Форма расчёта в CAD-стиле** (`/new/`):
- 3-панельная раскладка: hero · sidebar (параметры + список тормозов) · центральная панель (активный тормоз)
- Список тормозов в Project Explorer стиле с индикаторами активного и ошибок
- Кнопки на тормозе: «⊕ В каталог» (AJAX), «⎘ Копировать», «× Удалить»
- Имя тормоза синхронизируется живо с sidebar
- Кнопка «⏵ Запустить расчёт» в hero (через `form.requestSubmit()`)

**Срез 5 — Сравнение расчётов** (`/compare/`):
- 4 overlay-таба: x(t) · v·a(t) · v(x) · F_маг(v) — две кривые на одной оси (синий A vs розовый B)
- Дельта-таблица 12 метрик с цветными стрелками (▲ оранжевый = выросло, ▼ зелёный = уменьшилось)
- На дашборде: чекбоксы выбора + плавающий бар «Сравнить →»
- Лимит 2 расчёта одновременно, состояние в `sessionStorage`

## Архитектурные правила (соблюдай при изменениях!)

### Графики
- **ЛЮБОЙ** новый график → через хелперы из `recoil_app/services/charting.py`
- Константы: `RB_BLUE`, `RB_ACCENT`, `RB_GREEN`, `RB_AMBER`, `RB_PURPLE`, `RB_PINK`, `RB_GRAY`
- Заливки: `RB_BLUE_FILL`, `_CMP_FILL_A`, `_CMP_FILL_B`
- Толщины: `LINE_WIDTH_PRIMARY=3.0` (одиночные), `LINE_WIDTH_SECONDARY=2.5` (multi-curve), `LINE_WIDTH_DASHED=2.0`
- Шрифты: `FONT_FAMILY_UI` (Manrope), `FONT_FAMILY_MONO` (JetBrains Mono)
- Хелперы: `_apply_layout`, `_add_peak_marker`, `_add_recoil_vline`, `_series_color(i)`
- Маркеры пиков всегда справа: `ax=90, ay=0, arrowsize=1.4, arrowwidth=2`

### Шаблоны
- Все новые шаблоны наследуют `base_v2.html` (НЕ `base.html` — это старое)
- Топбар и левый рейл монтируются `shell.js` через `data-*` атрибуты
- Числовые значения через фильтр `{{ value|smart_num }}` (НЕ `floatformat:-6`)
- Для дат — `{{ ts|date:"d.m.Y · H:i" }}`

### Дизайн-система
- Файл: `recoil_app/static/recoil_app/css/design_system.css` (~1200 строк)
- CSS-переменные начинаются с `--rb-` (например `--rb-blue`, `--rb-panel-bg-white`)
- Классы компонентов: `rb-hero`, `rb-btn`, `rb-kpi`, `rb-tab`, `rb-stat`, `rb-form-section`, `rb-cad-*` etc.
- Hero-секция всегда сверху страницы, с `<div class="rb-color-bar">` (градиентная полоска)
- Кнопки: `rb-btn`, `rb-btn-primary`, `rb-btn-grad`, `rb-btn-danger`, `rb-btn-sm`
- Бейджи: `rb-badge`, `rb-badge-ok`, `rb-badge-warn`, `rb-badge-blue`, `rb-badge-accent`

### Модели и миграции
- Любые новые поля → через `python manage.py makemigrations`, никогда не правь схему вручную
- Текущая последняя миграция: `0018_brake_catalog`
- `MagneticBrakeConfig` и `BrakeCatalog` имеют 12 параметрических полей: `gamma, delta, n, xm, ym, dh1, dh2, dm, mu, bz, lya, wn0`
- Связь `BrakeCatalog → MagneticBrakeConfig` — copy-on-use (параметры копируются, без live-связи). Это правильно для воспроизводимости расчётов

### URL-схема
```
/                       → dashboard_view (имя 'dashboard')
/new/                   → index_view (имя 'index' — сохранено для обратной совместимости)
/run/<id>/              → run_detail_view (старая версия, не трогать)
/run/<id>/v2/           → run_detail_v2_view (новая, активная)
/run/<id>/delete/       → delete_run_view (POST)
/compare/               → compare_view
/catalog/               → catalog_list_view
/catalog/new/           → catalog_new_view
/catalog/<pk>/          → catalog_detail_view
/catalog/<pk>/edit/     → catalog_edit_view
/catalog/<pk>/delete/   → catalog_delete_view (POST)
/catalog/save-from-form/ → catalog_save_from_brake_form_view (AJAX POST)
```

### Физика расчёта (НЕ ТРОГАТЬ без обсуждения!)
- `services/dynamics.py` — численное интегрирование (Euler/RK4)
- `services/magnetic.py` — формула вихретокового тормоза
- `services/interpolation.py` — PCHIP + линейный хвост для табличных F(v)
- `services/analysis.py` — KPI, фазы, характерные точки

## Метод работы (важно!)

**Я работаю срезами:**
1. Перед каждым срезом я говорю что нужно (или ты предлагаешь, что осталось)
2. Ты **сначала задаёшь уточняющие вопросы и согласуешь план** — особенно если срез большой или меняет архитектуру
3. После моего «по твоему плану» / «делай» — реализуешь
4. Я тестирую у себя и присылаю скриншот / описание проблемы
5. Ты делаешь fix, я снова тестирую — итеративно до подтверждения «всё работает, идём дальше»

**Что важно:**
- Никаких больших переделок без согласования — лучше задать 5 вопросов, чем переделывать
- Каждое изменение должно быть проверяемо одной операцией: `python manage.py check`, `python manage.py runserver`, или открыть конкретную страницу в браузере
- Если файл больше 500 строк — не переписывай целиком, делай точечный `str_replace` или `edit`
- При работе с шаблонами всегда проверяй балансировку `{% if %}…{% endif %}`, `{% for %}…{% endfor %}`, `{% block %}…{% endblock %}`
- При работе с JavaScript всегда проверяй синтаксис через `node -e "new Function(...)"`
- При работе с Python всегда проверяй через `python -c "import ast; ast.parse(open('file.py').read())"`
